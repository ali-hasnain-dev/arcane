## 1. Distribution architecture (how this monorepo ships)

This repo (`arcane`, github.com/ali-hasnain-dev/arcane) is a PHP+JS monorepo.
Composer's `vcs` repository type only ever reads the `composer.json` at a
repo's root — it can't see packages living in subdirectories. So each
`packages/*` PHP package gets mirrored into its own standalone GitHub repo via:

- **`.github/workflows/split-packages.yml`** — on push to `main` (and tags),
  splits `packages/{support,actions,forms,tables,schemas,infolists,widgets,panels}`
  into repos named `larafusion-support`, `larafusion-actions`, etc. under
  `ali-hasnain-dev`, using `danharrin/monorepo-split-github-action@v2.4.5`
  (pin this version — v2.4.0 has a real crash bug on empty target repos, see
  `entrypoint.php`'s `execOrDie()` vs `exec()` in the checkout-branch logic).
  `packages/notifications` is JS-only (no composer.json), not split.
- Target repos need **one initial commit on `main`** before first split run
  (e.g. "Add a README") — a genuinely empty repo makes the action's own
  branch-bootstrap logic fail (`git push` with zero commits errors with
  `src refspec main does not match any`).
- Requires a `SPLIT_TOKEN` repo secret (classic PAT, `repo` scope) for the
  action to push into the target repos.
- **`composer.json` (repo root)** is now a real umbrella package
  (`larafusion/larafusion`, `"type": "library"`), requiring all seven
  sub-packages at `@dev` — same pattern as `filament/filament`. It still keeps
  its `path` repositories for local monorepo dev; those don't affect consumer
  installs via VCS.
- Full setup + consumer composer.json examples (full install vs.
  tables-only, etc.) are documented in **`docs/monorepo-split.md`**.
- Consumer install command (once repos + composer.json repositories are set
  up): `composer require larafusion/larafusion` then `php artisan larafusion:install`.

### Known install gotchas already solved
- GitHub API anonymous rate limit (60/hr) hits fast with 8 `vcs` repos —
  needs a PAT via `composer config -g github-oauth.github.com <token>`.
- **`vite.config.js`'s `larafusionResolve` plugin** (generated by
  `InstallCommand.php`) has two real bugs, both fixed in source:
  1. It only redirected *relative* cross-package imports; a lot of vendor
     source uses **bare** `@larafusion/support` (etc.) imports directly. Fixed
     by adding a `bareVendorMap` that resolves those bare specifiers straight
     to each vendor package's entry file.
  2. `resolveFile()`'s `existsSync(base)` check returns true for *directories*
     too, not just files — so it returned a bare directory path instead of
     ever checking `index.ts`. Fixed by requiring `statSync(base).isFile()`.
  - **These fixes only apply to fresh installs.** Anyone who ran
    `larafusion:install` before this fix has a stale `vite.config.js` that
    needs the same two edits applied by hand (the install command's own
    idempotency guard — checks for the string `'larafusion-cross-package-resolve'`
    — will skip rewriting an existing file).
- **`resources/css/app.css`'s `@source` patch** also had a silent-failure bug:
  it anchored on an assumed pre-existing `@source '...**/*.js';` line that
  doesn't exist in every Laravel/Tailwind v4 scaffold. `preg_replace()` with
  zero matches just returns the string unchanged — no error — so the critical
  `@source '../../vendor/larafusion/*/resources/js/**/*.{ts,tsx}'` line
  silently never got added, and the whole panel rendered unstyled. Fixed with
  a fallback anchor on `@import 'tailwindcss';` (always present) and a final
  append-if-nothing-matches fallback.
- `lucide-react@1.22.0` (installed here) doesn't export `Github` — Lucide
  dropped brand/logo icons from the core set at some point. Use
  `ExternalLink` or similar generic icons, not brand names, without checking
  `node_modules/lucide-react/dist/esm/icons/` first.

## 2. Dashboard (`packages/panels/resources/js/pages/Larafusion/Dashboard.tsx`)

- No hardcoded default widgets/stats. Removed the always-on `DeferredStats`
  block (3 stat cards) and the auto-generated resource nav-card grid.
- Two small static cards (greeting using `auth.user.name`, "Built with
  Larafusion" GitHub link) — genuinely static, no server round trip, no
  `<Deferred>`.
- Their visibility is controlled by a **named marker class**, not a boolean
  flag: `Larafusion\DefaultDashboardCards` (new file,
  `packages/panels/src/DefaultDashboardCards.php`). Presence in
  `->widgets([...])` on the Panel = cards show; absence = hidden. This is
  what the generated `PanelProvider` stub now does by default:
  `->widgets([DefaultDashboardCards::class])`. The marker gets filtered out
  in `LarafusionManager::registerPanel()` via `Panel::getRealWidgets()` before
  reaching `WidgetGrid`, so it's never treated as a real widget.
  `Panel::hasDefaultDashboardCards()` is the check; `DashboardController`
  passes it down as `showDefaultCards`.
- Cards sit in a `grid-cols-4` layout (not `grid-cols-2`) so they're compact
  (half a row), not full-width.

## 3. `php artisan larafusion:resource` scaffold (`MakeResourceCommand.php`)

- No longer auto-scaffolds `app/Larafusion/{Resources,Pages,Plugins}` or an
  example `Users` resource on `larafusion:install` — that's all gone from
  `InstallCommand.php`. Each of `larafusion:resource` /
  `larafusion:page` / `larafusion:plugin` / `larafusion:widget` creates its
  own directories on demand (verified each already does this).
- Generic (no-DB-introspection) fallback stubs (`genericFormStub`,
  `genericTableStub`) no longer guess `name`/`email`/`created_at` fields —
  empty arrays with commented-out examples instead.
- Generated `{Name}Resource.php` now overrides `canView(): bool { return false; }`
  when the developer declines the "generate a view page?" prompt. This
  matters because a generic `show()` route/page always works regardless of
  whether a custom `View{Name}` page was scaffolded — `canView()` was the
  only real lever to actually turn off the row-level View link/route.
- Generated `{Plural}Table.php` no longer calls `->filtersLayout()`,
  `->defaultSort(...)`, `->heading(...)` — all three already have sensible
  defaults elsewhere (`Table::$filtersLayout = 'dropdown'`,
  `ResourceController::index()` falls back to `id` asc,
  `Index.tsx` does `tableConfig?.heading ?? resource.navigationLabel`), so
  calling them in the stub was redundant. A comment above `configure()`
  documents how to override each one.

## 4. `Resource.php` / `ResourceController.php` / `Table.php` — response slimming

- Removed from the serialized `resource` node (`Resource::getPageProps()`):
  `hideCreateButton` (never read on frontend), `can` (whole object — auth is
  being split into its own package; frontend already falls back permissively
  everywhere it's read).
- Removed from index-page response specifically: `exportable`, `importable`
  (heads up: this does disable the Export/Import toolbar buttons in
  `BasicTable.tsx`, which read those two flags directly — accepted tradeoff
  per explicit user choice).
- `schema` prop on the **index** page only sent when
  `getInlineEditable()` is non-empty — omitted via array spread otherwise
  (not just sent as `[]`). Required adding `schema = []` as a destructuring
  default in `Index.tsx` (and making the type optional) since
  `flattenFields()` does a bare `for...of` with no undefined guard and would
  otherwise throw.
- **`showView`** re-added as its own top-level resource field (separate from
  the removed `can`) — this is a feature toggle (does this resource support a
  View link at all), not a permission, driven by the generated resource's
  `canView()` override above. `BasicTable.tsx`'s `can.view` is built from
  `resource.showView ?? true`, not from the removed `can` object.
- Column-narrowed `SELECT`: `ResourceController::index()` now only selects
  the DB columns the resource's configured `Table` columns actually need
  (plus primary key, `inlineEditable` fields, `deleted_at` if soft-deletes,
  record title attribute) — skipped entirely when any column is a relation
  (dot notation, e.g. `'author.name'`), since safely narrowing the local FK
  for every Eloquent relation type is out of scope. Documented limitation:
  a record action's `->visibleWhen()` closure reading a non-displayed
  attribute will see `null` now.
- `Table::pagination()` replaces `simplePagination()`/`disablePagination()`:
  `->pagination()` (full, default) / `->pagination('simple')` /
  `->pagination(false)`. Single `pagination: 'full'|'simple'|false` config
  key instead of two. Panel-level `->simplePagination()` (global default,
  different class) is untouched.
- `Table::poll()` renamed to **`polling()`** and actually implemented — it
  was computed/serialized but **never read anywhere in the frontend**
  before this session; same for `deferLoading()`. Both were pure no-ops.
  - `polling('30s')`: `BasicTable.tsx` parses the interval and runs
    `router.reload({only:['records']})` on a `setInterval`.
  - `deferLoading()`: `ResourceController::index()` wraps the initial
    `records` fetch in `Inertia::defer()` only when set; `Index.tsx` wraps
    the table in `<Deferred data="records">` with a `DeferredTable` +
    `TableSkeleton` (sized from the real `columns.length` and
    `resource.perPage` — added `perPage` to the resource node for this —
    header labels render as real text immediately, only body cells are
    skeletonized, since columns aren't deferred).
  - `striped()` was actually wired correctly; the real problem was
    (a) using `records.data.indexOf(record)` instead of the map index, and
    (b) a background tint too faint to notice. Fixed both.
- Row-actions column header ("Actions" text) is now blank —
  `BasicTable.tsx`.

## 7. Session: deferLoading skeleton, widget animations, create-button icon (2026-07-01)

Three changes:

1. **deferLoading skeleton keeps the search bar.** `Index.tsx` `TableSkeleton`
   now renders the real (disabled) search toolbar when `resource.searchable`
   is non-empty, and only skeletonizes the rows below the table headers.
   Both call sites now pass `resource`. Previously the whole table (incl.
   the toolbar/search) was replaced by a bare skeleton.

2. **Optional dashboard widget DATA animations.** Off by default. Animates the
   data *inside* each widget (NOT the card). Per type: stats count up + sparkline
   draws in; line/area **wipes in** L→R via an animated `<clipPath>` rect
   (scaleX, lf-wipe) with points popping behind; bars grow up (scaleY, fill-box,
   origin bottom); pie/doughnut **sweep** clockwise via `useReveal` progress
   (doughnut centre counts up in sync); radar/polar **grow from centre**
   (radius × progress); scatter/bubble pop. `useReveal(enabled)` is an rAF 0→1
   easeOutCubic hook for geometry that can't be a CSS keyframe. Both `useReveal`
   and `useCountUp` short-circuit under `prefers-reduced-motion` (JS reveals
   aren't covered by the CSS `[data-lf-anim]` guard).
   - PHP: `Panel::widgetAnimations(bool $enabled = true)`, property
     `$widgetAnimation` (bool), `getWidgetAnimation(): bool`, serialized as
     `widgetAnimation` in `toArray()` → `larafusion.panel.widgetAnimation`.
   - TS type: `PanelConfig.widgetAnimation?: boolean` (support types).
   - `packages/widgets/resources/js/lib/animations.ts` — `WidgetAnimationStyles`
     (`<style>` with keyframes lf-draw / lf-fade-in / lf-grow-up / lf-pop,
     reduced-motion guard on `[data-lf-anim]`) + `useCountUp(target, enabled)`
     rAF hook (easeOutCubic).
   - `WidgetGrid` takes `animate` (bool); `WidgetContent` passes `animated` to
     `StatsOverview` + `ChartWidget`. Plays once via `entered` state + 1300ms
     timer so polling doesn't re-count/redraw. `TableWidget` unchanged.
   - `StatsOverview`: `AnimatedStatValue` parses prefix/number/suffix + grouping
     and counts the numeric part up. `ChartWidget`: each sub-chart takes
     `animated` and applies inline SVG animation styles.
   - Wired from `Dashboard.tsx` (`animate={larafusion.panel.widgetAnimation}`).
   - NOTE: resource-index widgets (`Index.tsx` `DeferredWidgets`) is a
     pre-existing inconsistent/legacy path (uses `widgets` not `widgetsMeta`,
     omits `widgetDataUrl`) — left untouched. Animations wired on Dashboard only.
   - Docs: `docs/widgets.md` (Widget Animations — per-type table),
     `docs/panel-configuration.md`.

3. **Create button has no default icon.** Removed `'plus'` default from
   `Pages/Actions/CreateAction.php` (`$icon = null`) and from the controller
   fallback header action in `ResourceController::index()`. Users add one with
   `CreateAction::make()->icon('plus')`. `PageHeaderActions.tsx` already renders
   the icon only when present, so no JS change needed.

## 8. Session: column-driven table config (2026-07-01)

Filament-style move — the `table()->columns([...])` list is now the single source
of truth for search, inline-edit, relationship display, and what the query fetches.

- **`->searchable()` decoupled from filters.** `Column::searchable(bool)` now sets a
  real `$searchable` flag (global search), NOT `filterable()`. Removed the
  `searchable()` overrides in TextColumn/BooleanColumn/BadgeColumn/DateColumn that
  aliased to `filterable(<type>)`. **Behaviour change:** use `->filterable('boolean'|
  'select'|'date_range')` for per-column filters now. Added `Column::isSearchable()`.
- **`->inlineEditable()`** added on Column (+ `isInlineEditable()`). PHP method names
  are case-insensitive so `->inlineeditable()` also works.
- **Table builder** exposes `getSearchableColumnNames()` / `getInlineEditableColumnNames()`
  (mirrors existing `getSortableColumnNames()`).
- **Resource** `getSearchable()` = static `$searchable` ∪ column `->searchable()`;
  `getInlineEditable()` now derives from column `->inlineEditable()` (was `[]`).
  Both still overridable.
- **Relationship columns (dot notation, e.g. `category.name`) now work:**
  - ResourceController eager-loads relation paths (`->with(...)`, nested ok).
  - Base-table SELECT narrowed to local cols + key + inline + deleted_at + title +
    belongsTo FKs; skips narrowing for nested (`a.b.c`) → selects all base cols.
  - Search: dotted searchable cols use `orWhereHas(relation, attr like)`.
  - Sort: single-level belongsTo via correlated subquery `orderBy(Builder,...)`
    (no join); other relation types render but don't sort. Guarded in try/catch.
  - Frontend `BasicTable.tsx`: `resolveCellValue(record, 'category.name')` walks the
    dotted path (relation eager-loaded → `record.category.name`). Column label
    humanizes dots too → "Category Name".
  - `import BelongsTo` added to ResourceController.
- **Column-scoped fetch** = speed + security: undisplayed cols (e.g. `password`)
  never leave the DB.
- Scaffold: `MakeResourceCommand` drops `$searchable` from the resource stub, adds
  `->searchable()` to generated text columns, removed now-dead `buildSearchableArray()`,
  updated generic table-stub example comment.
- Docs: tables.md (Global Search / Inline Editing / Relationship Columns /
  Column-scoped queries sections + `->searchable()` semantics note), inline-editing.md,
  resources.md table, global-search.md.
- AdvancedTable.tsx (inactive phase-3 component) NOT updated for dotted paths — note
  if it's ever activated.
